1. TERMINOLOGY / NAMING OVERHAUL
--------------------------------------------------------------------------------
The core domain model was renamed throughout the entire stack (DB, Java, Angular)
to align with the agreed terminology:

  Old name                  →  New name
  ──────────────────────────────────────────────────────
  Implementation            →  Integration Method
  ImplementationVersion     →  ConnectorVersion
  ConnidVersion             →  (replaced by ConnectorBundle / ConnectorBundleVersion)
  ImplementationTag         →  ConnectorTag
  ImplementationImplementationTag → ConnectorConnectorTag
  ImplementationVersionLifecycleType (DB enum) → LifecycleType
  IntegrationMethodType (DB enum, fixed values) → integration_method_type table

  Upload form components    →  Publish form components
    upload-form-main        →  publish-form-main
    upload-form-impl        →  publish-form-impl

2. DATABASE SCHEMA CHANGES  (config/sql/01_schema.sql)
--------------------------------------------------------------------------------

2a. Renamed / changed enums
  - ImplementationVersionLifecycleType  →  LifecycleType  (same values)
  - IntegrationMethodType (fixed enum)  →  removed; replaced by a proper
    integration_method_type table (values loaded via seed data)
  - New enum: GlobalityType  ('GLOBAL' | 'SPECIFIC')  — for capability scope

2b. application table
  - Removed: last_modified, logo_content_type, logo_original_name, logo_size_bytes
  - Added:   updated (timestamp)
  - logo_path kept (other logo metadata moved to file system)

2c. Removed tables
  - implementation          (replaced by integration_method)
  - implementation_version  (replaced by connector_version)
  - connid_version          (replaced by connector_bundle / connector_bundle_version)

2d. New / changed tables

  connector_bundle
    - Existing table significantly expanded:
      display_name, description, ticketing_link, project_homepage,
      git_clone_url, path_to_project, build_framework (nullable),
      license (NOT NULL), framework (NOT NULL), lifecycle_state (NOT NULL)
    - build_framework is nullable — LOW_CODE connectors have no build tool

  connector_bundle_version  (new table)
    - Composite PK: (id, revision)
    - Tracks individual released versions of a bundle
    - Fields: bundle_version, browse_link, git_clone_url, path_to_project,
      build_framework (nullable), commit_tag, lifecycle_state

  connector  (new table)
    - Represents a single connector class inside a bundle
    - Fields: display_name, fully_qualified_class_name, connector_bundle_id,
      maintainer, author, description, revision

  connector_version  (new table)
    - Composite PK: (id, revision)
    - Links a connector to a specific bundle version
    - Fields: fully_qualified_class_name, lifecycle_state, connector_id,
      connector_bundle_version_id + revision

  connector_tag / connector_connector_tag
    - Renamed from implementation_tag / implementation_implementation_tag
    - Structure unchanged

  integration_method  (replaces implementation)
    - UUID primary key (composite with revision)
    - Fields: display_name, description, tutorial, file_path,
      midpoint_minVersion, midpoint_maxVersion, lifecycle_state,
      revision, author, maintainer, app_version, application_id

  integration_method_type  (new table, replaces fixed enum)
    - id, display_name, description
    - Seed data includes: ConnID Connector, REST API, SCIM, Manual/ITSM,
      Database, CSV, LDAP, Low-code Connector

  int_method_int_method_type  (new join table)
    - integration_method ↔ integration_method_type  (many-to-many)

  integration_method_connector  (new table)
    - Links an integration_method to a connector with version range
    - Fields: connector_id, connector_minVersion, connector_maxVersion

  capability  (new table)
    - id, name, description, display_order, globality (GLOBAL | SPECIFIC)
    - Seed data populates all ConnID capability types

  integration_method_capability / integration_method_capability_item
    - Stores capability groups per integration method (by object class)

  conn_version_capability / conn_version_capability_item
    - Stores capability groups per connector version (by object class)

  midpoint_version  (new table)
    - id, version, version_name, is_current
    - Used to select compatible midPoint version range for an integration method

  application_tag
    - Extracted into its own table (was previously managed differently)
    - Fields: id, name, display_name, tag_type (DEPLOYMENT|LOCALITY|CATEGORY|COMMON)

2e. Constraints & indexes
  - All new tables have proper PKs, FKs with ON DELETE CASCADE, and indexes
  - unique_connector_bundle_version on connector_bundle_version(bundle_version)
  - uq_imc_met_id_met_rev on integration_method_connector ensures one
    connector link per integration method version

3. NEW / CHANGED API ENDPOINTS  (GET /api/...)
--------------------------------------------------------------------------------

New endpoints:
  GET  /api/integration-method-types
       Returns all available integration method types (formerly a fixed enum)

  GET  /api/midpoint-versions
       Returns all midPoint versions for compatibility range selection

  GET  /api/upload/check-bundle-name?bundleName=…
       Returns true if a connector bundle name is already taken

  GET  /api/connectors/catalog
       Returns connector bundles in ACTIVE state for use in publish form
       connector picker (select existing connector when publishing)

  POST /api/integration-methods/search/{size}/{page}
       Searches integration methods with filtering (same shape as application search)

  POST /api/integration-methods/{id}/tutorial
       Uploads a tutorial file (PDF/XML/JSON/YAML/TXT) for an integration method

Changed endpoints:
  POST /api/upload/connector
       Payload restructured — now splits into three sub-objects:
         application, integrationMethod, connector
       connector sub-object carries: displayName, framework, buildFramework,
         gitCloneUrl, pathToProject, className, bundleName, bundleDisplayName,
         commitTag, connectorBundleId (for reusing an existing connector)
       Also accepts integrationMethodCapabilities and connectorCapabilities
       (capability groups per object class)

  GET  /api/capabilities
       Now returns full capability objects with globality (GLOBAL|SPECIFIC)
       instead of raw capability type strings

5. FRONTEND CHANGES  (Angular)
--------------------------------------------------------------------------------

5a. Components renamed
  upload-form-main  →  publish-form-main
  upload-form-impl  →  publish-form-impl
  Route updated:  /upload  →  /publish

5b. New component: capability-picker
  - Reusable picker for selecting capabilities per object class
  - Two sections: Global (applies to all object classes) and
    Specific (per named object class with add/remove rows)
  - Used in both publish-form-impl (connector capabilities) and request-form

5c. publish-form-impl (was upload-form-impl)
  - Step 5 now has three connector type paths:
      java-based         — full dev & build section (build tool, class name, repo)
      own-repo           — low-code with own git repository
      evolveum-hosted    — low-code, Evolveum creates the GitHub repo
  - "Select existing connector" path: connector picker loads catalog connectors
    and pre-populates all fields; skips bundle name / version uniqueness checks
  - Internal sub-steps: Add connector (5) → Compatibility (6) → Review (7)
  - Compatibility step: midPoint version range (min/max) per integration method
  - Connector capability groups sent alongside integration method capabilities
  - Tutorial file upload added (POST after main publish completes)
  - Build framework (MAVEN / GRADLE) only required for java-based connectors;
    sent as null for low-code (field is now nullable in DB)

5d. publish-form-main
  - Step labels updated to match new flow
  - Step 4 now shows connector type selector (java-based / own-repo /
    evolveum-hosted / existing catalog connector)

5e. request-form
  - Capability picker embedded in request form
  - Capabilities submitted as structured groups (objectClass + capability names)

5f. application-detail
  - Displays integration methods and their associated connector versions
  - Shows connector capabilities and midPoint version compatibility range
  - Download links per connector bundle version

5g. applications-list / filter-modal
  - Filtering updated to work with new integration_method and tag structure
  - Filter by integration method type (previously hardcoded enum, now dynamic)

5h. Models (new/changed)
  - catalog-connector.model.ts    — new; represents a catalog connector entry
  - application-detail.model.ts   — updated to reflect new nested structure
    (IntegrationMethod → ConnectorVersion → ConnectorBundleVersion)
  - implementation-list-item.model.ts — updated field names
  - request.model.ts              — capabilities now use structured groups

5i. Auth — login and logout now trigger window.location.reload() so that
  per-role connector visibility is refreshed immediately

7. NOTES FOR MIGRATION (existing database)
--------------------------------------------------------------------------------
  - This branch is a full DB rebuild — the old schema is NOT forward-compatible
  - To migrate an existing instance: drop and recreate the DB from the new
    01_schema.sql + 02_data.sql scripts
  - Two columns changed from NOT NULL to nullable:
      connector_bundle.build_framework
      connector_bundle_version.build_framework
    If applying to an existing new-schema DB:
      ALTER TABLE connector_bundle ALTER COLUMN build_framework DROP NOT NULL;
      ALTER TABLE connector_bundle_version ALTER COLUMN build_framework DROP NOT NULL;
